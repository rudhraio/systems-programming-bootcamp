# Learner Context

## Current Knowledge

The learner knows basic C:

- variables
- functions
- loops
- arrays
- structs
- basic pointers

The learner is not starting from zero, but should not be assumed to know advanced C, modern C++, operating systems, linkers, build systems, or Linux internals.

## Development Setup

Host operating system:

- macOS

Editors:

- Vim
- VS Code

Compilers:

- Clang
- GCC

Shell:

- zsh
- bash

Debuggers:

- LLDB on macOS
- GDB in Linux environment when available

Linux practice should use one of:

- UTM Linux VM
- Docker
- VMware Fusion
- Parallels
- remote Linux machine

## Available Time

Exactly one hour per day.

Daily lessons must be small enough to complete without spillover:

```text
20 min learning
35 min coding
 5 min revision
```

## Long-Term Goals

The learner wants to become highly comfortable with:

- C
- modern C++
- systems programming
- Linux programming
- build systems
- toolchains
- memory
- debugging
- eventually Linux device driver development

## Strengths

- Already knows C basics
- Has a working development environment
- Wants production-quality understanding
- Is interested in long-term depth instead of quick interview preparation

## Likely Weaknesses

- May not yet have strong mental models for memory layout
- May confuse pointer values, pointed-to objects, and ownership
- May not know how compilers, assemblers, linkers, and loaders cooperate
- May not know the difference between macOS and Linux system interfaces
- May not yet debug from evidence using tools
- May need repeated exposure to build systems and command-line workflows

## Preferred Learning Style

Use:

- practical examples
- ASCII diagrams
- memory diagrams
- terminal commands
- small projects
- debugging tasks
- revision checklists

Avoid:

- long theory dumps
- excessive historical background
- unexplained jargon
- interview-only tricks
- large projects before the mental model is ready

## Explanation Structure

For new concepts:

1. Why this exists
2. What problem it solves
3. How it behaves mechanically
4. Small code example
5. Command-line inspection
6. Common bugs
7. Practice task

## Tone

Direct, practical, and rigorous.

The mentor should be encouraging through clarity and useful feedback, not through empty praise.
